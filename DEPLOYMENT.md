# Развёртывание Ai Arbuz Provider Api

Сервис представляет собой stateless-приложение, которое слушает **внутренний
порт** и рассчитано на работу **за обратным прокси** (TLS + домен). Это позволяет
ему мирно сосуществовать с любыми сервисами, которые уже работают на хосте —
в том числе с другим reverse-proxy (Caddy, Nginx, Traefik), который уже занимает
порты 80/443.

---

## 1. Принцип

```
[ Клиенты ] → HTTPS → [ Ваш обратный прокси :443 ] → HTTP → [ Arbuz :ВНУТРЕННИЙ_ПОРТ ]
```

- Приложение **не** должно публиковать 80/443 само.
- Приложение слушает локальный/внутренний порт (`ARBUZ_LISTEN`, по умолчанию
  `:8080`). Этот порт можно сменить на любой свободный.
- TLS, домены и маршрутизацию выполняет ваш обратный прокси.

---

## 2. Предварительные требования

- Docker + Docker Compose (рекомендуемый способ) **или** собранный бинарь Go.
- Обратный прокси с TLS (любой):
  - уже работающий на сервере — просто добавьте новый сайт/маршрут; **или**
  - поднимите новый, если его ещё нет.
- Домен/субдомен, указывающий на сервер.

---

## 3. Подготовка окружения

```bash
cp .env.example .env
```

Заполните как минимум:

- `ARBUZ_MASTER_KEY` — 32 байта (`openssl rand -hex 32`). **Не меняйте после
  первого запуска** — иначе зашифрованные секреты станут нечитаемыми.
- `ARBUZ_ADMIN_PASSWORD` — надёжный пароль (пустой/слабый/placeholder приложение
  отклонит на старте).
- `ARBUZ_LISTEN` — внутренний порт. **Если `:8080` занят — поставьте свободный**,
  например `:8090`.
- `ARBUZ_PUBLIC_URL` — ваш публичный адрес, напр. `https://api.example.com`.

---

## 4. Запуск через Docker Compose

Чтобы избежать конфликтов портов, публикуйте порт приложения **только на
localhost** (а наружу его отдаёт обратный прокси):

```yaml
services:
  arbuz:
    # ...
    ports:
      - "127.0.0.1:8090:8080"   # ХОСТ(127.0.0.1):ВНУТРЕННИЙ -> порт контейнера
    env_file: .env
    volumes:
      - ./data:/app/data        # персистентное хранилище БД/секретов
    restart: unless-stopped
```

```bash
docker compose up -d
docker compose logs -f arbuz
```

> Левая часть `127.0.0.1:8090` — свободный порт на хосте; меняйте его при
> конфликте. Правая часть (`8080`) — это `ARBUZ_LISTEN` внутри контейнера.

### Если ваш прокси сам в Docker
Удобнее подключить приложение к общей docker-сети прокси и вообще **не
публиковать порт на хост** — прокси будет ходить к контейнеру по имени сервиса:

```yaml
services:
  arbuz:
    # без секции ports
    networks: [proxy_net]
networks:
  proxy_net:
    external: true   # сеть, в которой уже живёт ваш Caddy/Nginx/Traefik
```

Тогда апстрим в прокси указывается как `http://arbuz:8080`.

## 5. Конфигурация обратного прокси

### Caddy

```caddyfile
api.example.com {
    reverse_proxy 127.0.0.1:8090      # или http://arbuz:8080 при общей docker-сети
}
```

Caddy сам получит и продлит сертификат. Перезагрузка: `caddy reload` (или
`docker exec <caddy> caddy reload --config /etc/caddy/Caddyfile`).

> Если Caddy уже обслуживает другой сайт — просто **добавьте новый блок** для
> вашего поддомена. Ничего из существующего трогать не нужно.

### Nginx

```nginx
server {
    server_name api.example.com;
    location / {
        proxy_pass         http://127.0.0.1:8090;
        proxy_http_version 1.1;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;

        # Потоковая выдача (SSE): отключаем буферизацию
        proxy_buffering    off;
        proxy_read_timeout 1h;
    }
    # listen 443 ssl; + сертификаты (например, через certbot)
}
```

### Traefik
Навесьте на сервис лейблы роутера для вашего домена и укажите внутренний порт
`8080` как loadbalancer-порт; entrypoint — ваш websecure.

## 6. Проверка

```bash
curl -fsS https://api.example.com/api/health
```

Затем откройте `https://api.example.com` и войдите в панель.

---

## 7. Эксплуатация

- **Данные:** всё хранится в томе `./data` (БД + зашифрованные секреты). Бэкапьте
  его; не коммитьте в git (исключён в `.gitignore`).
- **Логи:** `docker compose logs -f arbuz` (структурированный JSON).
- **Обновление:**
  ```bash
  git pull
  docker compose build
  docker compose up -d
  ```
- **Смена порта при конфликте:** правьте левую часть `ports` в compose
  и/или `ARBUZ_LISTEN`, затем `docker compose up -d`.

---

## 8. Чек-лист безопасности

- [ ] Надёжные `ARBUZ_ADMIN_PASSWORD` и `ARBUZ_MASTER_KEY`.
- [ ] Порт приложения **не** доступен из интернета напрямую (только через прокси).
- [ ] TLS терминируется на прокси, валидный сертификат.
- [ ] Том `data` бэкапится, `.env` не попадает в репозиторий.
- [ ] `ARBUZ_LIMIT_FAIL=closed` (если важна защита расходов на апстрим).
